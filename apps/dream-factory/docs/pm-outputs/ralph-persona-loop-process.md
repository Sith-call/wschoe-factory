# Ralph Persona Loop — 유저 페르소나 기반 앱 개선 프로세스

## 목적
타겟 세그먼트 내 다양한 유저 관점으로 앱을 반복 개선하여, 여러 유형의 유저가 모두 80% 이상 만족하는 앱을 만든다.

## 프로세스

```
┌─────────────────────────────────────────────┐
│  1. PM: 페르소나 생성                         │
│     - 타겟 세그먼트 내에서만 (이탈 금지)        │
│     - 이전 페르소나와 다른 관점/성격/니즈        │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│  2. 페르소나 평가                              │
│     - 스크린별 워크스루 (페르소나 시점)          │
│     - 5개 차원 점수: 첫인상/사용성/몰입/공유/재방문 │
│     - 종합 만족도 산출                         │
│     - Top 3 페인포인트 + Top 3 딜라이트         │
└──────────────────┬──────────────────────────┘
                   ▼
              만족도 >= 80%?
              /          \
           Yes            No
            │              │
            ▼              ▼
┌──────────────────┐  ┌─────────────────────────┐
│  3a. 커밋          │  │  3b. PM 가설 수립         │
│  - 페르소나명+점수  │  │  - "If X, Then Y, Because Z" │
│  - 변경사항 요약    │  │  - 우선순위 정렬           │
│  - git commit      │  └────────────┬────────────┘
└────────┬─────────┘               ▼
         │              ┌─────────────────────────┐
         │              │  4. Design+Dev 병렬 개선    │
         │              │  - 가설 기반 구현           │
         │              │  - 빌드 검증               │
         │              └────────────┬────────────┘
         │                           │
         │                           ▼
         │                    2번으로 돌아가기
         │                    (동일 페르소나 재평가)
         ▼
┌─────────────────────────────────────────────┐
│  5. 새 페르소나 생성                           │
│     - 같은 세그먼트, 다른 관점                  │
│     - 이전 페르소나들의 니즈와 겹치지 않는 특성    │
│     → 2번으로 돌아가기                         │
└─────────────────────────────────────────────┘
```

## 규칙

1. **세그먼트 이탈 금지**: 타겟 유저 그룹 밖의 페르소나는 만들지 않는다
2. **정직한 평가**: 페르소나는 개발자/디자이너가 아닌 실제 유저로서 판단한다
3. **가설 기반 개선**: PM은 피드백만으로 자체 가설을 세우고, 이를 적용한다
4. **커밋 = 마일스톤**: 80% 도달 시 반드시 커밋하여 진전을 기록한다
5. **누적 개선**: 이전 페르소나의 만족을 깨뜨리지 않으면서 새 페르소나도 만족시킨다

## 팀 역할

| 팀 | 역할 |
|----|------|
| **PM** | 페르소나 생성, 평가 실행, 가설 수립, 우선순위 결정 |
| **Design** | 비주얼/UX 관련 가설 구현 (Stitch 수정 또는 직접 UI 개선) |
| **Dev** | 기능/로직/데이터 관련 가설 구현, 빌드 검증 |

## 평가 차원

| 차원 | 설명 | 가중치 |
|------|------|--------|
| 첫인상 | 앱이 신뢰감 있고 매력적으로 보이는가? | 20% |
| 사용성 | 설명 없이 사용할 수 있는가? | 25% |
| 재미/몰입 | 재미있는가? 계속 쓰고 싶은가? | 25% |
| 공유욕구 | 친구에게 보여주고 싶은가? | 15% |
| 재방문 의향 | 내일 다시 올 것인가? | 15% |
