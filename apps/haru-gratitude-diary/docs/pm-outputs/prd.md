# haru-gratitude-diary

**하루 감사 일기 -- 매일 감사한 일 3가지를 기록하고 주간 회고로 돌아보는 최소주의 감사 일기 앱**

- **Version:** 1.0 (MVP)
- **Date:** 2026-04-10
- **Status:** Draft
- **Author:** PM Executor

---

## 개요

매일 감사한 일 3가지를 30초 안에 기록하고, 주간 회고로 내가 소중히 여기는 것을 발견하는 최소주의 감사 일기 앱.

**핵심 가치 제안:** "3줄 쓰고, 1주일 돌아보고, 나를 발견한다."

| 항목 | 내용 |
|------|------|
| 타겟 | 25-34세 한국 여성 직장인 (primary) |
| 플랫폼 | 웹앱 (모바일 최적화) |
| 비즈니스 모델 | Free Core + Optional Premium (2,900원/월) |
| 북극성 지표 | 주간 회고를 완료한 사용자 수 (Weekly Active Reflectors) |
| MVP 화면 수 | 5개 |
| 작성 완료 목표 시간 | 30초 이내 |

## 문제 정의

### 사용자가 겪는 핵심 고통

1. **습관 붕괴 (Forgetting):** 감사 일기를 시작해도 5-7일 후 알림이 무시되고 습관이 형성되지 않는다. 맥락적 트리거 없이는 신경 경로가 충분히 강화되지 않는다.

2. **반복 피로 (Repetition Fatigue):** 첫 주가 지나면 매일 같은 항목(커피, 날씨, 식사)을 쓰게 된다. 일기가 의미 있는 행위가 아니라 기계적 반복으로 느껴지면서 7일-30일 사이에 이탈이 급증한다.

3. **강제 긍정에 대한 거부감 (Forced Positivity):** 힘든 날에 억지로 감사를 찾는 것은 "앱 형태의 독성 긍정"으로 느껴진다. 스트릭을 깨지 않으려고 억지로 쓰다가 결국 앱 자체를 삭제한다.

4. **빈 페이지 불안 (Blank Page Anxiety):** "무엇에 감사하나요?"라는 열린 질문은 결정 피로를 유발한다. 빈 텍스트 필드를 30초 바라보다 부적절감을 느끼고 앱을 닫는다.

5. **보이지 않는 보상 (No Visible Payoff):** 매일 감정 노동을 투입하지만 돌아오는 것이 없다. 축적된 기록이 패턴을 보여주거나 기분을 전환하는 "아하 모먼트"가 존재하지 않는다. 일기는 쓰기만 하고 다시 꺼내 보지 않는 블랙홀이 된다.

6. **기능 과잉 (Feature Bloat):** 기존 감사 앱들은 비전보드, 확언, 무드 트래커, 소셜 기능을 번들한다. "그냥 감사 일기만 쓰고 싶은" 사용자는 압도당한다.

### 현재 대안과 한계

| 대안 | 한계 |
|------|------|
| 종이 노트 | 검색 불가, 주간 요약 불가, 빈 페이지 막막함, 3개월 전 기록은 사실상 소실 |
| Gratitude (Gratefulness.me) | 비전보드, 확언, 커뮤니티 등 기능 과잉. 프리미엄 유도 팝업. |
| Five Minute Journal | 영어 전용, 한국어 미지원. 연 $40 구독. 주간 회고 없음. |
| Day One | 범용 일기 앱. 감사 구조 없음. 빈 페이지 = 종이 노트와 동일 문제. |
| 마보, 마인드카페 | 명상/상담 중심. 감사 일기는 부수 기능이거나 없음. 앱 자체가 무거움. |
| 데일리 다이어리, 오늘도 감사 | 감사 특화이나 주간 회고 약함. 반복 피로 해결책 없음. |

### 왜 지금인가

- 한국 시장에 "감사 특화 + 최소주의" 포지션을 차지한 앱이 없다 (blue ocean).
- COVID-19 이후 "마이크로 웰니스" 수요가 지속적으로 증가하고 있다.
- 구독 피로가 심화되면서 "핵심 무료 + 선택적 프리미엄" 모델에 대한 수용도가 높아졌다.
- 한국어로 감정을 기록하고 싶은 욕구가 영어 앱으로는 충족되지 않는다.

## 화면 구성

MVP는 5개 화면으로 구성한다.

### 1. Home (홈)

- **역할:** 앱의 메인 화면. 오늘의 기록 상태를 보여주고, 핵심 액션으로 안내.
- **주요 요소:**
  - 오늘 날짜 표시
  - 오늘 기록 완료 여부 (미완료: CTA 버튼 "오늘의 감사 쓰기" / 완료: 오늘 쓴 3가지 요약)
  - 이번 주 기록 현황 (7일 중 N일 기록)
  - 주간 회고 진입 버튼 (일요일 또는 7일 데이터 존재 시 활성)
  - 하단 내비게이션: 홈 / 캘린더

### 2. Entry (감사 기록)

- **역할:** 오늘의 감사 3가지를 입력하는 핵심 화면.
- **주요 요소:**
  - 오늘 날짜
  - 3개의 텍스트 입력 필드 (플레이스홀더: 부드러운 프롬프트 힌트)
  - 저장 버튼
  - 뒤로가기 (홈으로)
- **설계 원칙:** 앱을 열고 3줄 쓰고 저장까지 30초. 다른 요소 일절 없음.

### 3. Calendar (캘린더)

- **역할:** 월별 기록 현황을 시각적으로 확인.
- **주요 요소:**
  - 월별 캘린더 그리드
  - 기록한 날: 채움 표시 (색상 도트)
  - 미기록 날: 빈 상태 (죄책감 없는 중립적 디자인)
  - 날짜 탭 시 해당 일의 Detail 화면으로 이동
  - 월 이동 (이전/다음)
  - 이번 달 기록 통계: "N일 기록"

### 4. Detail (일별 상세)

- **역할:** 특정 날짜의 감사 기록을 확인하고 수정.
- **주요 요소:**
  - 해당 날짜
  - 감사 3가지 목록
  - 수정 버튼 (인라인 편집)
  - 삭제 버튼 (확인 다이얼로그)
  - 뒤로가기 (캘린더 또는 홈으로)

### 5. Weekly Reflection (주간 회고)

- **역할:** 일주일 치 감사를 모아서 패턴을 보여주는 핵심 차별화 화면.
- **주요 요소:**
  - 해당 주의 날짜 범위 (예: 4/1 - 4/7)
  - 일주일간 기록한 모든 감사 항목 목록 (날짜별 그룹)
  - 반복 키워드 하이라이트 (빈도 기반)
  - "이번 주 인사이트" 한 줄 요약 (예: "이번 주, 당신은 '사람'에게서 가장 많은 감사를 느꼈어요")
  - 기록한 날수 / 7일
  - 홈으로 돌아가기

## 기술 스택

| 영역 | 기술 | 근거 |
|------|------|------|
| 프레임워크 | React 18+ | 컴포넌트 기반 UI, 생태계 |
| 언어 | TypeScript | 타입 안전성, 유지보수성 |
| 빌드 도구 | Vite | 빠른 개발 서버, 최적 빌드 |
| 스타일링 | Tailwind CSS | 유틸리티 기반, 빠른 UI 구현 |
| 상태 관리 | React Context + useReducer | 외부 라이브러리 불필요한 규모 |
| 데이터 저장 | localStorage | 서버 불필요, 프라이버시 보장, 제로 인프라 비용 |
| 라우팅 | React Router v6 | SPA 내비게이션 |
| 날짜 처리 | date-fns | 경량, 트리쉐이킹 가능 |

**아키텍처 원칙:**
- 백엔드 없음. 모든 데이터는 브라우저 localStorage에 저장.
- 인증 없음. 가입/로그인 절차 없이 즉시 사용.
- 데모 모드 필수. 앱 최초 실행 시 샘플 데이터 없이 바로 빈 상태에서 시작하되, 주간 회고 체험을 위한 데모 데이터 생성 옵션 제공.

## 성공 지표

### 북극성 지표 (North Star Metric)

**"주간 회고를 완료한 사용자 수 (Weekly Active Reflectors)"**

이 지표가 높다는 것은 (1) 일주일간 꾸준히 감사를 기록했고, (2) 주간 회고까지 완료했다는 뜻이므로, 핵심 가치(감사 습관 형성 + 회고를 통한 자기 발견)를 모두 반영한다.

2. 주간 회고를 본 사용자의 다음 주 리텐션이 보지 않은 사용자보다 높은가?
3. D7 리텐션이 50%를 넘기는가?

## 제약 사항

| 제약 | 설명 |
|------|------|
| 데모 모드 필수 | DB 없이 localStorage만으로 모든 기능이 동작해야 한다. 앱을 열면 바로 체험 가능해야 한다. |
| 백엔드 없음 | 서버 API, 데이터베이스, 인증 서비스를 사용하지 않는다. |
| 인증 없음 | 가입, 로그인, 소셜 로그인 일절 없음. |
| 한국어 전용 | MVP는 한국어만 지원. 다국어는 추후 고려. |
| 모바일 퍼스트 | 데스크탑도 동작하되, 모바일 웹 기준으로 설계. |
| localStorage 용량 | 브라우저 localStorage 한도(약 5-10MB) 내에서 동작해야 한다. 1년 치 데이터(365일 x 3항목) 기준 충분하나, 장기 사용 시 데이터 내보내기로 대응. |
| 오프라인 미지원 (MVP) | Service Worker 기반 오프라인 모드는 MVP 범위 밖. |
