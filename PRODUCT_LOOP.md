# Product Loop v2 — 프로덕트 제작 루프

> v1 루프에서 10개 앱을 만들며 발견한 문제점을 반영한 개선 버전.
> 핵심 변화: 디자인 먼저, 피드백과 구현 분리, 학습 자동 축적.

---

## 루프 개요

```
Phase 0: 시장 조사         (병렬 3 에이전트, ~5분)
    ↓
Phase 1: 기획              (병렬 2 에이전트, ~8분)
    ↓
Phase 2: 디자인 시스템      (1 에이전트, ~5분)
    ↓
Phase 3: 개발              (1-2 에이전트, ~5분)
    ↓
Phase 4: PM 검증           (1 에이전트, ~3분)
    ↓
Phase 5: QA 테스트         (1 에이전트, ~3분)
    ↓
Phase 6: Ralph Loop ★      (반복 루프, 전원 통과까지)
    ┌────────────────────────────────────────────┐
    │  Step A: 피드백 수집 (병렬 5)               │
    │    ├─ 페르소나 1 → 만족도 점수              │
    │    ├─ 페르소나 2 → 만족도 점수              │
    │    ├─ 페르소나 3 → 만족도 점수              │
    │    ├─ Design Visionary → 디자인 점수        │
    │    └─ PRD Verifier → PRD 반영도 점수 ← NEW │
    │              ↓                             │
    │  Step B: 전원 80%+ AND 디자인 7+            │
    │          AND PRD P0 반영 90%+?       ← NEW │
    │    ├─ YES → 루프 종료                      │
    │    └─ NO  ↓                                │
    │  Step C: 통합 구현 (1 에이전트)              │
    │    (PRD 갭 → 디자인 → UX 순으로 우선)       │
    │              ↓                             │
    │  → Step A로 (다음 Iteration)                │
    └────────────────────────────────────────────┘
    ↓
Phase 7: 룰 업데이트        (자동, ~1분)
    ↓
Phase 8: 배포 준비
```

**총 예상 시간: ~45분 + Ralph 이터레이션당 ~15분**
**총 에이전트: ~12개 + Ralph 이터레이션당 5개**

## Phase 0: 시장 조사

### 목적
앱 컨셉을 데이터 기반으로 결정. 감이 아니라 시장에서 답을 찾는다.

### 실행
3 에이전트 병렬:
1. **생태계 분석** — 타겟 플랫폼의 기존 앱, 인기 앱, 포화/블루오션 카테고리
2. **기술 역량 조사** — SDK/API 제약, 백엔드 필요 여부, 사용 가능한 모듈
3. **트렌드 리서치** — MZ세대 패턴, 바이럴 공식, 경쟁 앱 분석

### 산출물
- 시장 조사 리포트 (3개 에이전트 결과 종합)
- 추천 앱 컨셉 TOP 3 (블루오션 점수 + 금융 시너지 + 바이럴 잠재력 + 기술 실현성)

### 종료 조건
유저가 컨셉 1개를 선택하면 다음 단계로.

## Phase 1: 기획

### 목적
프로덕트 레벨의 깊이 있는 기획. "뭘 만들지"가 아니라 "왜, 누구를 위해, 어떻게"까지.

### 실행
2 에이전트 병렬:

**PM 에이전트:**
- PRD (문제 정의 + 핵심 가치 + 3 페르소나 + MVP 기능 + 성공 지표)
- 유저 스토리 맵 (시간축: 첫 방문 → 1주 → 1달, 15+ 스토리)
- 화면 플로우 (전체 화면 목록 + 네비게이션 구조 + 화면별 상세)

**기술 아키텍처 에이전트:**
- TRD (데이터 모델 + API 설계 + SDK 모듈 계획)
- 프로젝트 구조
- 배포 계획

### 산출물
```
docs/pm-outputs/prd.md
docs/pm-outputs/user-stories.md
docs/pm-outputs/screen-flows.md
docs/trd.md
```

### 종료 조건
PRD + TRD 파일이 생성되고 빌드 가능한 프로젝트 구조가 정의되면.

## Phase 2: 디자인 시스템 (NEW)

### 목적
개발 전에 디자인 언어를 확정. "코드 짜고 나서 예쁘게" 가 아니라 "디자인 먼저, 코드는 따르기".

### v1의 문제
- 개발 에이전트가 임의로 색상/폰트를 결정
- Design Visionary가 사후에 "이건 AI 같다"고 지적 → 대규모 수정 필요
- 같은 실수를 반복 (중앙 정렬, 카드 네스팅, 모노 컬러)

### 실행
1 에이전트 (Design Visionary):
1. DESIGN_RULES.md 읽기
2. 기존 앱들의 사용된 팔레트/폰트 확인 (중복 방지)
3. 앱 성격에 맞는 디자인 토큰 결정:
   - 색상 팔레트 (primary, secondary, surface, text, accent, danger)
   - 폰트 조합 (한국어 + 영문)
   - border-radius 전략 (어디에 sharp, 어디에 rounded)
   - 버튼 스타일 (primary vs secondary 구분)
   - 레이아웃 원칙 (좌측 정렬 vs 중앙, 카드 네스팅 금지)
   - 애니메이션 전략 (어디에 즉시, 어디에 전환)
4. 핵심 화면 3개의 와이어프레임 (ASCII or Stitch)

### 산출물
```
docs/design-system.md
```
이 파일에:
- 색상 토큰 (hex 값)
- 폰트 토큰 (CDN URL 포함)
- 컴포넌트 스타일 (버튼, 카드, 입력 필드)
- 레이아웃 규칙
- DESIGN_RULES.md 체크리스트 통과 확인

### 종료 조건
design-system.md가 작성되고, DESIGN_RULES.md 체크리스트 전항목 통과.

## Phase 3: 개발

### 목적
PRD + design-system.md + TRD를 따라서 구현.

### v1의 문제
- 개발 에이전트가 PRD를 "참고"만 하고 자기 판단으로 구현
- 디자인 토큰이 없어서 임의 색상/폰트 사용
- 유저 스토리 23개 중 실제 구현된 것이 몇 개인지 불명확

### 실행
1-2 에이전트 (복잡도에 따라):
1. **반드시** 이 파일들을 먼저 읽기:
   - `docs/pm-outputs/screen-flows.md` (화면 설계)
   - `docs/design-system.md` (디자인 토큰)
   - `docs/trd.md` (기술 아키텍처)
   - `DESIGN_RULES.md` (안티패턴 룰)
2. 화면 플로우대로 모든 화면 구현
3. design-system.md의 토큰을 그대로 사용 (색상/폰트 임의 변경 금지)
4. `npm run build` 통과 확인

### 산출물
- 전체 소스 코드 (`src/`)
- 빌드 통과 확인

### 종료 조건
`npm run build` 에러 0.

## Phase 4: 초기 PM 검증

### 목적
개발 직후 치명적인 기획-개발 갭을 빠르게 잡는다. Ralph 루프 전에 명백한 누락을 먼저 수정.

### 실행
1 에이전트 (빠른 체크):
1. `docs/pm-outputs/user-stories.md`의 P0 스토리만 확인
2. 라이브 앱에서 P0 기능 존재 여부 체크
3. **명백한 누락만** 잡는다 (세부 품질은 Ralph 루프에서)
4. 누락된 P0 기능이 있으면 → Phase 3으로 돌아가 구현

### 종료 조건
P0 유저 스토리의 핵심 기능이 "존재"하면 통과. 품질은 Ralph에서 검증.

> **참고**: 상세한 PRD 반영도 검증은 Phase 6 Ralph 루프의 PRD Verifier가 매 이터레이션마다 수행한다.

## Phase 5: QA 테스트 (NEW)

### 목적
기능적 버그, 엣지 케이스, 접근성 문제를 잡는다.

### v1의 문제
- "눈으로 스크린샷 확인"이 QA의 전부였음
- 엣지 케이스 (빈 데이터, 긴 텍스트, 오프라인 등) 미검증
- 접근성 (폰트 크기, 터치 타겟, 컬러 대비) 미검증

### 실행
1 에이전트 (browse tool 활용):
1. 전 화면 모바일 뷰포트(375x812) 스크린샷
2. 핵심 유저 플로우 end-to-end 테스트
3. 엣지 케이스 테스트:
   - 빈 데이터 상태 (첫 방문)
   - 극단적 입력 (모든 최고, 모든 최저)
   - 뒤로가기 플로우
   - 새로고침 후 상태 복원
4. 접근성 체크:
   - 최소 폰트 사이즈 13px
   - 최소 터치 타겟 44px
   - 컬러 대비 4.5:1 이상
5. 콘솔 에러 확인

### 산출물
```
docs/qa-report.md
```
- 화면별 테스트 결과
- 발견된 버그 목록 (severity: critical/high/medium/low)
- 접근성 체크 결과

### 종료 조건
Critical/High 버그 0개.

## Phase 6: Ralph Loop (반복 루프)

### 목적
**페르소나 + 디자인 비저너리의 만족도가 모두 80% 이상이 될 때까지 반복.**
"기능이 작동하냐"가 아니라 "이 사람들이 진짜 만족하냐".

### v1의 문제
- Ralph가 1회만 돌고 끝남 (피드백 → 구현 → 완료)
- 구현 후 실제로 만족도가 올랐는지 재검증 없음
- 디자인 비저너리가 Ralph와 별도로 돌아서 타이밍 안 맞음

### 루프 구조

```
┌────────────────────────────────────────────────────┐
│                  Ralph Iteration N                  │
│                                                    │
│  Step A: 피드백 수집 (병렬 5 에이전트)                │
│    ├─ 페르소나 1 (UX 피드백 + 만족도 점수)            │
│    ├─ 페르소나 2 (UX 피드백 + 만족도 점수)            │
│    ├─ 페르소나 3 (UX 피드백 + 만족도 점수)            │
│    ├─ Design Visionary (디자인 크리틱 + 점수)        │
│    └─ PRD Verifier (PRD 반영도 검증 + 점수)  ← NEW  │
│         ↓                                           │
│  Step B: 판정                                       │
│    페르소나 전원 ≥ 80%                               │
│    AND 디자인 ≥ 7/10                                │
│    AND PRD 반영도 ≥ 90%?                     ← NEW  │
│    ├─ YES → 루프 종료                               │
│    └─ NO  → Step C로                               │
│         ↓                                           │
│  Step C: 통합 구현 (1 에이전트)                      │
│    ├─ 5개 피드백 리포트 읽기                         │
│    ├─ 우선순위 정리 + 충돌 해소                     │
│    ├─ 한 번에 모든 개선사항 구현                     │
│    ├─ npm run build + 라이브 검증                   │
│    └─ 반영 체크리스트 작성                          │
│         ↓                                           │
│  → Step A로 돌아가기 (Iteration N+1)                │
│                                                    │
└────────────────────────────────────────────────────┘
```

### Step A: 피드백 수집

**5 에이전트 병렬** (코드 수정 금지!):

**페르소나 에이전트 3명:**
1. 각 에이전트는 배정된 페르소나로 앱을 사용
2. browse tool로 전 화면 라이브 테스트
3. **피드백 리포트만 작성** (코드 수정 절대 금지)
4. 산출물:
   - 화면별 피드백 (좋은 점 + 개선점)
   - 개선 제안 3-5개 (우선순위: P0/P1/P2)
   - **만족도 점수** (0-100%)
   - 만족도 근거 (왜 이 점수인지)

**Design Visionary 1명:**
1. 375x812 모바일 뷰포트로 전 화면 스크린샷
2. DESIGN_RULES.md 기준 + 추가 디자인 기준으로 크리틱
3. AI 안티패턴 체크 (중앙 정렬, 카드 네스팅, 모노 컬러, 버튼 위계 등)
4. 산출물:
   - 화면별 디자인 크리틱
   - 개선 제안 3-5개
   - **디자인 점수** (1-10)

**PRD Verifier 1명:**
1. `docs/pm-outputs/prd.md`와 `docs/pm-outputs/user-stories.md` 읽기
2. `docs/pm-outputs/screen-flows.md`의 화면 설계 읽기
3. 라이브 앱을 browse tool로 전 화면 확인
4. 유저 스토리별 구현 여부 체크:
   ```
   US-1.1: 앱 열면 3초 안에 가치 이해 → ✅ 구현 / ❌ 미구현 / ⚠️ 부분
   US-2.4: 결과에 레이더 차트 표시 → ✅ 구현
   US-3.2: 챌린지 규칙 + 예상 절약 미리보기 → ⚠️ 부분 (금액만 있고 규칙 설명 부족)
   ```
5. 화면 플로우 대비 누락 화면 체크
6. 디자인 시스템(`design-system.md`) 준수 여부 확인 (색상/폰트/레이아웃 토큰)
7. 산출물:
   - P0 유저 스토리 구현율 (%)
   - 전체 유저 스토리 구현율 (%)
   - 누락된 화면/기능 리스트
   - 디자인 토큰 위반 사항
   - **PRD 반영도 점수** (0-100%)

### 산출물 (Iteration N별)
```
docs/ralph/iteration-N/persona-1.md
docs/ralph/iteration-N/persona-2.md
docs/ralph/iteration-N/persona-3.md
docs/ralph/iteration-N/design-visionary.md
docs/ralph/iteration-N/prd-verification.md
```

### Step B: 판정

**통과 조건 (모두 충족해야 함):**
- 페르소나 1 만족도 ≥ 80%
- 페르소나 2 만족도 ≥ 80%
- 페르소나 3 만족도 ≥ 80%
- 디자인 점수 ≥ 7/10
- PRD P0 반영도 ≥ 90%

**하나라도 미달이면** → Step C로 이동, 개선 후 재피드백.

**우선순위 (Step C에서):**
1. PRD 미구현 (P0) → 기능이 없으면 페르소나 만족도도 올라가지 않음
2. 디자인 안티패턴 → AI 냄새 제거
3. 페르소나 피드백 → UX 개선

**최대 반복 횟수**: 5회. 5회째에도 미달이면 현재 상태로 출시하고 피드백을 docs에 기록.

### Step C: 통합 구현

**1 에이전트** (피드백-구현 분리 원칙):
1. Step A의 5개 피드백 리포트 전부 읽기
2. 피드백 통합 정리:
   - PRD 미구현 항목 먼저 (기능 갭이 만족도의 근본 원인)
   - 디자인 안티패턴 다음
   - 페르소나 UX 피드백 마지막
   - 중복 피드백 병합, 충돌 시 PRD > 디자인 > UX 순
   - P0 피드백만 이번 이터레이션에서 구현 (P1/P2는 다음 이터레이션)
3. 한 번에 모든 개선사항 구현
4. `npm run build` 통과 확인
5. 라이브 앱에서 개선사항 스크린샷 검증
6. 반영 체크리스트 작성:
   ```
   [페르소나1 #1] 퀴즈 격려 메시지 → ✅ 반영
   [페르소나2 #3] 카테고리별 금액 → ✅ 반영
   [디자인 #2] 좌측 정렬 변경 → ✅ 반영
   [페르소나3 #4] 일일 팁 → ⚠️ P1, 다음 이터레이션
   ```

### 종료 조건
**모든 평가자가 통과 기준을 충족하면 루프 종료.**

### 예상 시나리오

**Iteration 1:**

## 핵심 원칙

### 1. 디자인 먼저 (Design-First)
개발 에이전트는 `design-system.md`의 토큰을 따른다. 색상/폰트를 임의로 정하지 않는다.

### 2. 피드백-구현 분리 (Separation of Concerns)
Ralph는 피드백만 한다. 코드를 고치지 않는다. 통합 에이전트가 한 번에 구현한다.

### 3. 학습 축적 (Learning Accumulation)
매 루프에서 발견한 안티패턴은 DESIGN_RULES.md에 자동 추가된다. 같은 실수는 2번 하지 않는다.

### 4. 검증이 있어야 넘어간다 (Gate-Based Flow)
각 Phase에 종료 조건이 있다. "빌드 되니까 OK"가 아니라 "PM 스토리 100% 구현" + "Critical 버그 0개" + "디자인 점수 7+"를 충족해야 다음으로.

### 5. 병렬화 가능한 건 병렬로 (Maximize Parallelism)
시장 조사 3개, 기획 2개, Ralph 3개는 항상 동시 실행. 순차 의존성이 있는 것만 순차로.
