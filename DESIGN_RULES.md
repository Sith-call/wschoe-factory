# AI 안티패턴 방지 디자인 룰

> 이 문서는 9개 앱 코드 분석(82개 fadeInUp, 57개 135deg gradient, 55개 #fbbc00 인스턴스)에서
> 도출한 규칙이다. 새 앱을 만들 때 반드시 따를 것.

---

## 핵심 원칙

**"모든 앱이 같은 AI가 만든 것처럼 보이면 안 된다."**

각 앱은 독립된 제품이다. 색상, 폰트, 레이아웃, 애니메이션, 네비게이션 구조가
다른 앱과 달라야 한다. "이쁘게"가 아니라 "이 앱답게" 만들어라.

---

## 금지 규칙 (NEVER)

### R1. 색상 복붙 금지
- `#fbbc00` + `#111417` 조합을 새 앱에서 사용하지 마라.
- 기존 앱과 동일한 Tailwind color config를 복사하지 마라.
- 새 앱마다 **고유한 primary/secondary/surface 팔레트**를 정해야 한다.
- 팔레트를 정할 때 앱의 성격을 반영하라:
  - 게임/재미 → 비비드, 컬러풀 (예: 핫핑크, 라임, 시안)
  - 생산성 → 미니멀, 무채색 + 1 액센트
  - 웰니스 → 뮤트 파스텔 또는 자연색
  - 금융 → 네이비, 그린, 화이트

### R2. 그라디언트 텍스트 금지
- `-webkit-background-clip: text` + gradient 조합을 헤딩에 쓰지 마라.
- `.text-gradient` 클래스를 만들지 마라.
- 텍스트는 **단색**으로. 강조가 필요하면 굵기, 크기, 색상으로 구분하라.

### R3. 135deg 그라디언트 남용 금지
- `linear-gradient(135deg, ...)` 를 앱 전체에서 3회 이상 쓰지 마라.
- 그라디언트를 쓸 거면 각도를 의도적으로 선택하라 (180deg, 90deg, radial 등).
- 버튼 + 제목 + 카드 + 배지 전부에 같은 그라디언트를 넣지 마라.

### R4. fadeInUp 만능 금지
- `translateY(20px) → translateY(0)` 를 기본 애니메이션으로 쓰지 마라.
- 한 앱에서 **2가지 이상 다른 entrance 패턴**을 써라:
  - 좌/우에서 슬라이드
  - 스케일 (0.95 → 1)
  - 즉시 표시 (애니메이션 없음!)
  - 스프링/바운스 이징
- 모든 요소에 entrance animation을 넣지 마라. **대부분은 즉시 표시**가 맞다.

### R5. 동일 네비게이션 금지
- 모든 앱에 `fixed bottom-0` 탭바를 넣지 마라.
- 앱 구조에 맞는 네비게이션을 선택하라:
  - 선형 플로우 (퀴즈, 온보딩) → 탭바 없이 뒤로가기만
  - 2-3 섹션 → 상단 세그먼트 컨트롤 또는 스와이프
  - 4+ 섹션 → 바텀 탭바 허용, 단 디자인은 앱마다 다르게
  - 단일 화면 → 네비게이션 없음

### R6. backdrop-blur 남발 금지
- 다크 배경 위 다크 컴포넌트에 `backdrop-blur`를 넣지 마라 (시각적 효과 0, 성능만 낭비).
- 실제로 뒤의 콘텐츠가 비쳐 보이는 상황에서만 사용하라.
- 한 앱에서 backdrop-blur 사용은 **최대 2곳** (예: 모달 오버레이, 스티키 헤더).

### R7. 장식 파티클/오브 금지
- 떠다니는 점, 블러 오브, 별 배경을 자동으로 넣지 마라.
- 장식 요소는 **기능적 의미가 있을 때만** (예: 축하 컨페티는 OK, 배경 파티클은 NO).
- 빈 공간(화이트 스페이스)이 장식보다 강력하다.

## 다양성 규칙 (MUST VARY)

### R8. 앱마다 다른 폰트
- "Plus Jakarta Sans + Manrope + Noto Sans KR" 조합을 또 쓰지 마라.
- 한국어 폰트 옵션: Pretendard, Spoqa Han Sans, IBM Plex Sans KR, Nanum Gothic
- 영문 폰트는 앱 성격에 맞게: Inter, DM Sans, Space Grotesk, Outfit, Sora 등
- 시스템 폰트 (`font-family: system-ui`)도 좋은 선택이다.

### R9. 다크 모드 일변도 탈피
- 전체 앱의 50% 이상이 다크 모드면 새 앱은 **라이트 모드**로 만들어라.
- 라이트 모드 ≠ 촌스러움. 잘 만든 라이트 테마가 차별화 요소가 된다.
- 화이트 + 1색 액센트 조합은 매우 효과적.

### R10. rounded-2xl 균일화 금지
- 모든 카드에 같은 border-radius를 넣지 마라.
- 위계에 따라 radius를 다르게:
  - 메인 컨테이너: 큰 radius (16-24px) 또는 0
  - 내부 요소: 작은 radius (4-8px)
  - 버튼: 앱마다 다르게 (pill, rounded, sharp 중 택 1)
- 한 앱 안에서도 **의도적으로 sharp corner를 섞어라**.

### R11. 아이콘 다양화
- Material Symbols Outlined를 모든 앱에서 쓰지 마라.
- 대안: Phosphor Icons, Heroicons, Lucide, 커스텀 SVG
- 한 앱 안에서 아이콘 세트를 섞지 마라 (하나로 통일).

### R13. 이모지를 디자인 요소로 쓰지 마라
- 이모지(🔥🐔💰 등)를 아이콘, 장식, UI 요소로 사용하지 마라.
- 이모지는 OS/기기마다 렌더링이 다르고, 프로페셔널한 앱처럼 보이지 않는다.
- AI가 "쉽고 빠른 아이콘"으로 이모지를 남발하는 것 자체가 안티패턴이다.
- 대신 사용할 것:
  - SVG 아이콘 라이브러리 (Lucide, Phosphor, Heroicons)
  - 커스텀 일러스트 또는 단색 SVG
  - 텍스트 + 색상 조합으로 시각적 구분
- 예외: 유저가 직접 입력하는 콘텐츠 (예: 채팅, 리액션)에서는 이모지 OK.

### R14. 전부 중앙 정렬 금지
- 인트로/랜딩을 제외한 모든 화면에서 **좌측 정렬을 기본**으로 하라.
- AI는 모든 것을 중앙 정렬하는 경향이 있다. 이것이 가장 큰 "AI 냄새" 중 하나다.
- 좌측 정렬이 읽기 편하고 프로페셔널하다.
- 예외: 결과 화면의 메인 숫자, 로딩 화면의 메시지 정도만 중앙 정렬 허용.

### R15. 카드 네스팅 금지
- 카드 안에 카드를 넣지 마라 (카드 → 배경 → 카드 → 배경의 층).
- 배경과 같은 색의 카드는 시각적으로 의미가 없다.
- 구분이 필요하면 border-left 액센트, 구분선, 또는 여백으로 해결하라.
- "카드를 넣으면 예뻐 보인다"는 AI의 착각이다.

### R16. 버튼 위계 필수
- Primary 액션과 Secondary 액션의 스타일이 달라야 한다.
- Primary: 솔리드 배경 (다크 또는 primary 색상) + 흰 텍스트
- Secondary: 아웃라인 (테두리만) 또는 텍스트 버튼
- 두 버튼이 같은 스타일이면 유저는 뭘 눌러야 하는지 모른다.
- 공유/부가 기능 같은 tertiary 액션은 텍스트 링크로 충분하다.

### R17. 로딩 화면 목적감
- 스피너 + 재미 텍스트 로테이션은 AI 앱의 전형이다.
- 대신 체크리스트형 로딩 (단계별로 완료 표시) 또는 프로그레스 바를 사용하라.
- 유저가 "지금 뭐 하고 있는지" 알 수 있어야 한다.

### R12. heading에 font-extrabold 도배 금지
- 모든 제목에 `font-extrabold` (800)을 쓰지 마라.
- 제목 위계: H1만 bold/extrabold, H2는 semibold, H3는 medium
- 가벼운 폰트 웨이트(300, 400)를 대비 요소로 활용하라.

## 체크리스트

새 앱을 만들기 전에 확인:

```
[ ] 기존 앱과 다른 색상 팔레트인가?
[ ] 기존 앱과 다른 폰트 조합인가?
[ ] 그라디언트 텍스트를 안 쓰는가?
[ ] 네비게이션 패턴이 앱 구조에 맞는가?
[ ] fadeInUp 외에 다른 (또는 없는) 애니메이션을 쓰는가?
[ ] 불필요한 장식 요소(파티클, 오브)가 없는가?
[ ] backdrop-blur를 의미 있는 곳에만 쓰는가?
[ ] 다크/라이트 테마 비율이 균형 잡혀 있는가?
[ ] 이모지를 아이콘/장식으로 쓰지 않았는가?
[ ] 콘텐츠가 좌측 정렬 기본인가? (중앙 정렬 남용 X)
[ ] 카드 안에 카드를 넣지 않았는가?
[ ] Primary/Secondary 버튼 스타일이 구분되는가?
[ ] 로딩 화면이 목적감 있는가? (스피너+텍스트 X)
```

## 좋은 예시 vs 나쁜 예시

### 색상
```
BAD:  primary: "#fbbc00", surface: "#111417"  (또 같은 팔레트)
GOOD: primary: "#2563eb", surface: "#fafafa"  (앱 성격에 맞는 고유 팔레트)
```

### 애니메이션
```
BAD:  모든 카드에 animate-fade-in-up class
GOOD: 첫 번째 카드는 즉시 표시, 두 번째는 100ms 딜레이, 세 번째는 200ms (시차 로딩)
```

### 버튼
```
BAD:  <button class="btn-gradient rounded-full px-8 py-4">  (매번 동일)
GOOD: <button class="bg-blue-600 text-white rounded-lg px-5 py-2.5 font-medium">  (앱에 맞게)
```

### 네비게이션
```
BAD:  퀴즈 앱인데 fixed bottom-0 탭바 3개
GOOD: 퀴즈 앱은 상단 프로그레스 바 + 뒤로가기만
```

### 아이콘
```
BAD:  <span className="text-4xl">💰</span>  (이모지를 아이콘으로)
GOOD: <svg>...</svg> 또는 Lucide 아이콘 라이브러리 사용
```
