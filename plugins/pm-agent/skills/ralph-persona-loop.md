# Ralph Persona Loop

PM이 만든 유저 페르소나가 앱을 평가하고, 디자이너가 디자인 통일성을 검토한 뒤, 유저 만족도 80% + 디자인 통일성 80%에 도달할 때까지 Design+Dev 팀이 반복 개선하는 프로세스. 한 페르소나가 만족하면 커밋 후 새 페르소나로 전환하여 앱을 점진적으로 발전시킨다.

## 전체 플로우

```
Phase 1: 페르소나 생성 (PM)
    ↓
Phase 1.5: QA 테스트 (app-qa-tester)
    ├─ PASS → Phase 2로
    └─ FAIL → Dev 수정 → QA 재테스트
    ↓
Phase 2: 유저 평가 (페르소나 시점)
    ↓
Phase 2.5: 디자이너 평가 (디자인 통일성)
    ↓
Phase 3: 분기 판단
    ├─ 유저 80%+ AND 디자인 80%+ → Phase 5 (커밋)
    └─ 그 외 → Phase 4 (개선)
                ├─ PM: 가설 수립 (유저 피드백 + 디자인 리뷰 종합)
                ├─ Design팀: Stitch 수정 → HTML 재다운로드
                └─ Dev팀: 기능 개선 + 디자인 싱크 → 빌드
                    ↓
                Phase 1.5로 복귀 (QA → 재평가)
Phase 5: 커밋 (페르소나명 + 점수)
    ↓
Phase 6: 새 페르소나 (같은 세그먼트, 다른 관점) → Phase 1로
```

## 왜 이 프로세스가 중요한가

개발자/디자이너의 관점과 실제 유저의 관점은 다르다. "잘 만들었다"와 "쓰고 싶다"는 전혀 다른 기준이다. 페르소나 기반 테스트는 출시 전에 이 간극을 줄여주고, 다양한 유저 유형의 시각을 미리 반영할 수 있게 한다. 커밋 단위로 기록하면 어떤 개선이 어떤 유저 유형을 만족시켰는지 추적할 수 있다.

## 전제 조건

- 앱이 빌드 완료되어 데모 가능한 상태
- Stitch 디자인 레퍼런스 또는 스크린샷이 존재
- PM 산출물 (user-stories.md, prd.md) 존재

## Phase 1: 페르소나 생성

PM이 타겟 세그먼트 내에서 유저 페르소나를 정의한다.

**산출물**: `docs/pm-outputs/persona.md` (또는 `persona-N.md`)

포함 항목:
- 이름, 나이, 직업, 성격 (MBTI 등)
- 기술 친숙도 (1-5)
- 일상적으로 사용하는 앱 목록
- 이 앱을 시도하게 된 동기
- 삭제 트리거 (어떤 상황에서 삭제할지)
- 공유 트리거 (어떤 상황에서 공유할지)

**규칙**:
- 타겟 세그먼트 밖의 유저는 만들지 않는다
- 이전 페르소나와 다른 관점/성격/니즈를 가져야 한다
- 실제로 이 앱을 다운로드할 법한 사람이어야 한다

## Phase 1.5: QA 기능 검증

유저 피드백을 받기 전에, 기본 기능이 정상 작동하는지 먼저 검증한다. 깨진 기능에 대한 피드백은 UX 개선에 도움이 되지 않는다.

**에이전트**: `dev-team:app-qa-tester`

**프로세스**:
1. PM의 user-stories.md에서 인수 조건(AC) 추출
2. 각 AC를 테스트 케이스로 변환
3. 코드 읽기 기반 논리 검증 + 빌드 검증 + 스크린샷 검증
4. Pass/Fail 리포트 작성

**산출물**: `docs/pm-outputs/iteration-N-qa-report.md`

**분기**:
- P0 블로커 또는 P1 3개 이상 → **FAIL** → Dev팀이 수정 후 QA 재테스트
- P1 1-2개 이하 → **PASS** → Phase 2 (페르소나 평가)로 진행

이 게이트가 있어야 페르소나가 "버튼이 안 눌려요" 대신 "이 흐름이 좀 답답해요" 같은 진짜 UX 피드백에 집중할 수 있다.

## Phase 2: 페르소나 평가

페르소나가 되어 앱을 스크린별로 워크스루하고 평가한다.

**평가 방법**:
1. 스크린 레퍼런스 이미지(Stitch PNG 또는 라이브 스크린샷)를 순서대로 확인
2. 각 스크린에서 페르소나 시점의 솔직한 반응을 기록 (한국어, 자연스러운 말투)
3. 5개 차원 점수 매기기
4. 종합 만족도 산출

**평가 차원**:

| 차원 | 설명 | 가중치 |
|------|------|--------|
| 첫인상 | 앱이 신뢰감 있고 매력적으로 보이는가? | 20% |
| 사용성 | 설명 없이 사용할 수 있는가? | 25% |
| 재미/몰입 | 재미있는가? 계속 쓰고 싶은가? | 25% |
| 공유욕구 | 친구에게 보여주고 싶은가? | 15% |
| 재방문 의향 | 내일 다시 올 것인가? | 15% |

**종합 만족도** = 각 차원 점수 x 가중치의 합

**산출물**: `docs/pm-outputs/iteration-N-feedback.md`

포함 항목:
- 페르소나 요약 (1문단)
- 스크린별 반응 (페르소나 말투로, 한국어)
- 5개 차원 점수 + 종합 만족도
- Top 3 페인포인트
- Top 3 딜라이트
- PM 가설 (다음 Phase에서 사용)

**솔직함 원칙**: 페르소나는 개발자가 아닌 실제 유저다. "이게 뭐지?", "좀 답답하네", "오 이거 좋다" 같은 자연스러운 반응을 사용한다. 기술적 용어를 쓰지 않는다.

## Phase 2.5: 디자이너 평가

유저 페르소나 평가 직후, 디자이너가 전문적 시각으로 앱의 디자인을 평가한다. 유저는 "뭔가 이상해"라고 느끼지만 정확히 뭐가 문제인지 모르는 경우가 많다. 디자이너는 그 "이상함"의 정확한 원인을 진단한다.

**디자이너 평가 차원**:

| 차원 | 설명 | 가중치 |
|------|------|--------|
| 컬러 통일성 | 스크린 간 색상 팔레트가 일관되는가? | 25% |
| 타이포그래피 | 폰트 사용이 일관되고 가독성이 좋은가? | 20% |
| 레이아웃 일관성 | 여백, 간격, 컴포넌트 크기가 통일되는가? | 20% |
| 인터랙션 패턴 | 선택/전환/네비게이션 패턴이 일관되는가? | 15% |
| 브랜드 아이덴티티 | 앱 전체에서 하나의 브랜드로 느껴지는가? | 20% |

**디자이너 산출물**: `docs/pm-outputs/iteration-N-design-review.md`

포함 항목:
- 스크린별 디자인 이슈 (구체적 문제점 + 스크린 간 비교)
- 디자인 통일성 점수 (0-100)
- 디자인 개선 제안 (구체적: "IntroScreen의 버튼 라운드가 28px인데 EmotionScreen에서는 16px")
- Stitch 수정이 필요한 스크린 목록
- 긴급도 분류: 🔴 통일성 파괴 / 🟡 개선 권장 / 🟢 양호

**PM과의 합의**: 디자이너 리뷰가 완료되면 PM과 디자이너가 다음을 결정한다:
1. 유저 피드백과 디자인 이슈 중 겹치는 문제 (최우선 해결)
2. 유저는 못 느꼈지만 디자인적으로 문제인 것 (이번에 같이 해결할지 다음으로 미룰지)
3. Stitch에서 스크린 재생성/수정이 필요한 범위

## Phase 3: 분기 판단

두 가지 점수를 모두 확인:

```
유저 만족도 >= 80% AND 디자인 통일성 >= 80% → Phase 5 (커밋)
그 외 → Phase 4 (개선)
```

유저 만족도가 80%여도 디자인 통일성이 낮으면 개선이 필요하다. 디자인 부채는 시간이 지나면 유저 만족도도 떨어뜨리기 때문이다.

## Phase 4: 가설 기반 개선

PM이 유저 피드백 + 디자이너 리뷰에서 개선 가설을 도출하고, Design+Dev 팀이 병렬로 구현한다.

### PM 가설 수립

각 가설은 이 형식을 따른다:

```
### H{N}: {가설 제목}
> **If** {구체적 변경 사항}을 적용하면,
> **Then** {차원} 점수가 {현재} → {목표}로 상승할 것이다.
> **Because** {유저 심리/행동 근거}.
```

가설을 impact 순으로 정렬하고, 상위 3-5개만 이번 이터레이션에서 실행한다.

### 팀 배분

| 가설 유형 | 담당 팀 | 실행 순서 |
|----------|---------|----------|
| 디자인 통일성/스크린 수정 | Design 팀: Stitch 수정 → HTML 재다운로드 | 1단계 |
| 비주얼/UX 코드 개선 + 디자인 싱크 | Dev 팀: Stitch HTML 기준 React 매칭 | 2단계 (Design 완료 후) |
| 기능/로직/데이터/텍스트 개선 | Dev 팀 | 1단계 (Design과 병렬 가능) |

### Design 팀 워크플로우 (디자인 개선이 필요한 경우)

디자이너 리뷰에서 Stitch 수정이 필요하다고 판단된 스크린에 대해:

1. **Stitch 스크린 수정**: `mcp__stitch__edit_screens`로 기존 스크린 수정하거나, `mcp__stitch__generate_variants`로 대안 탐색
2. **HTML 재다운로드**: 수정된 스크린의 HTML 소스를 다시 다운로드
3. **레퍼런스 스크린샷 갱신**: Playwright로 430x932에서 새로 렌더링
4. **핸드오프 문서 업데이트**: 변경된 디자인 토큰, 클래스, 아이콘 반영

### Dev 팀 워크플로우

기능 개선과 디자인 싱크를 함께 수행:

1. **기능 개선**: PM 가설에 따른 로직/데이터/텍스트 변경
2. **디자인 싱크**: Design 팀이 갱신한 Stitch HTML을 React 컴포넌트에 반영 — Tailwind 클래스 복사, 커스텀 CSS 업데이트
3. **빌드 검증**:
```bash
npx tsc --noEmit  # TypeScript 검증
npm run build     # 프로덕션 빌드
```

빌드 실패 시 수정 후 다시 시도.

### Phase 2로 복귀

동일 페르소나로 재평가 + 디자이너 재검토. 두 점수 모두 올랐는지 확인.

## Phase 5: 커밋 (80% 도달 시)

만족도 80% 이상이면 현재 상태를 커밋한다.

**커밋 메시지 형식**:
```
Persona "{이름}" satisfied ({점수}%)

- {주요 변경사항 1}
- {주요 변경사항 2}
- {주요 변경사항 3}

Iteration: {이터레이션 수}
Dimensions: 첫인상 {n}/사용성 {n}/몰입 {n}/공유 {n}/재방문 {n}
```

**이터레이션 기록 업데이트**: `docs/pm-outputs/ralph-persona-loop-process.md`의 기록 테이블에 행 추가.

## Phase 6: 새 페르소나로 전환

커밋 후 PM이 새 페르소나를 생성한다.

**새 페르소나 설계 원칙**:
- 같은 타겟 세그먼트 내에서 선택
- 이전 페르소나와 다른 관점 (나이, 성격, 동기, 기술 수준 등)
- 이전 페르소나의 만족을 깨뜨리지 않으면서 새로운 관점을 테스트

**예시 다양성 축**:
- 나이대 (20대 vs 30대 vs 40대)
- 기술 친숙도 (파워유저 vs 캐주얼)
- 동기 (재미 vs 자기탐색 vs 공유)
- 성격 (꼼꼼한 vs 성급한 vs 탐색적)

Phase 1로 돌아가서 반복.

## 누적 개선 원칙

여러 페르소나를 거치면서 앱이 발전할 때, 이전 페르소나의 만족을 깨뜨리지 않아야 한다. 새 페르소나를 위한 변경이 기존 강점을 훼손하는지 항상 확인한다. 트레이드오프가 발생하면 PM이 판단하여 가중치를 조정한다.
